# lrnev 产品战略笔记（2026-06-12）

> 来源：与 Claude 的产品方向讨论快照，scene `01-findings-remediation` 评审之后；同日经 GPT 交叉评审收敛（维护态缺口定性修正、顺序依赖钉死，见各节标注）。
> 性质：内部战略思考，非执行清单。落地前每一步仍需单独评估、按 lrnev 流程开 spec。
> 前置共识：scene 01（确定性硬校验等 7 个 spec）是地基，先做完；本文讨论的是地基之后往哪走。

## 主线：把「交税」变成「分红」

lrnev 的根本处境：用户前期交流程税（写 requirements、过 gate、建 task），但结构化数据的红利只兑现了一小部分（接手快照、追溯靠人翻）。

**发展主线一句话：让攒下的结构化数据持续产出回报。回报越大，用户越愿意交税，入口依从性问题反而自己缓解一半。**

每一步都在回答「用户为什么继续用」，而不是「还能加什么功能」。

---

## 两个生死问题（比任何功能都优先）

### 问题一：维护态缺口（决定留存）

项目越成熟，工作越是小增量（给已完成特性加参数、修边角），lrnev 的流程却越不贴身。这不是边角问题——**小增量是大多数真实工作的形态**。缺口不补，用户会在度过初建期后慢慢绕开。

**为什么现有轻产物（memory/errorbook/ADR）接不住**：它们是**知识轨道**（记「这件事教会了我们什么」），不是**工作轨道**（记「这件事做没做完、对应哪个需求」）。典型场景「给已完成的登录功能加个记住我选项」：errorbook 没踩坑、memory 不是约定、ADR 没有决策、新开 spec 写不出独立验收——AI 真机表现为乱开新 spec（2026-06-05 xpaas-gen-v3 观察）。

**定性修正（2026-06-12，GPT 交叉评审 + 核对 `docs/GOVERNANCE-FLOW.md` 后）：不是「没家可放」，是「有家但 AI 找不到门」。**

协议层的维护通道**已经存在**：状态机明文允许 `completed -> in-progress` 回退；GOVERNANCE-FLOW.md 原话「只是增量加需求时，在本版 task_create 即可，不必新开 spec」，且「需求落位」一节已把 context_search 落位定义为 AI 的判断职责。真实缺口是**发现性**：

1. 这条路写在 `docs/GOVERNANCE-FLOW.md` 里，AI 会话中不会读它；
2. 决策时刻 AI 面前的分流指引（CLAUDE.md 规则、工具 followup）是二元的——「开 spec 还是直接做」，没有第三分支「落位到现有 spec」；
3. 「先 context_search 落位再动手」没有任何工具层提醒在那个时刻把它顶到 AI 面前。

**方向（v1 = 铺显已有路径，不发明新实体、不加 flag）**：纯 followup/steering 工程——`spec_create` 的 followup 加「先 context_search 确认是否该落位到现有 spec」；给 completed spec 加 task 时自动提示状态回退语义（`completed -> in-progress`）。`--maintenance` flag、`change_record` 新实体都属于轻度过度设计，**真机验证「路显性了 AI 还是不走」之后才考虑动 API**。scene 01 实施过程本身会产出维护态标本（completed spec 的后续小修补），是 v1 设计的第一手需求素材。

**关键设计约束：必须比「直接改代码」只多一步操作，否则没人用。**

### 问题二：入口依从性（决定价值下限）

硬校验只能拦「已经进了流程的坏数据」，拦不住「AI 压根不开 spec 直接改代码」。当前最外层入口靠 CLAUDE.md 式规则——提示词载体有三个结构性缺陷：

1. **注意力衰减**：会话越长权重越稀释（I-19 已观察到 AI 对常驻 warning 脱敏，同一机制作用于入口规则）。
2. **执行时机错位**：规则在会话开头注入，但需要在「准备改代码的那一刻」被想起。
3. **lrnev 是被动方**：MCP server 只能看见发给它的调用；AI 完全不调它时，连 warning 的机会都没有。

**解法：执法点搬到动作发生的地方，三道环（按通用性排序，见下节修正）**：

| 环 | 载体 | 时机 | 通用性 |
|---|---|---|---|
| git 卡点 | pre-commit 配方：diff 动了源码但无 task 活动 → 提醒（默认）/拒绝（严格模式可选） | 提交时 | ✅ 100% 客户端无关、模型无关 |
| 事后审计 | doctor/report 加项：对比近期 git 提交与 task 状态变更，列出「未治理的变更」 | 任意时刻 | ✅ 住在 lrnev 体内 |
| harness 钩子 | SessionStart 注入治理快照、PostToolUse 查 in_progress task 并注入提醒 | 实时 | ✗ 按客户端，做薄垫片 |

**git 环的安装悖论（2026-06-12 实现者评审指出，落地前必须定策略）**：pre-commit hook 要先装进用户仓库才生效——安装动作本身就是一个入口依从性问题，且是全链路侵入性最高的一步（动用户 git 配置）。但注意它与 AI 依从性问题有本质差别：**hook 安装是人类的一次性动作，AI 依从是每会话的持续行为**——前者恰好落在「主动选择的绕过」框架里（装不装是一次知情决定），难度低一个量级。落地策略倾向：**`lrnev integrate git` 显式命令安装（与 integrate <client> 同一模式），init 绝不自动装、只在 followup 提一句**；配方需探测既有 hook 管理器（husky / core.hooksPath）并链式追加而非覆盖，不与用户已有钩子打架。

**顺序硬依赖（不可交换）**：doctor「未治理变更」审计命中的恰恰大多是维护态小改动。若审计环先上、维护通道还没铺显，审计会对无处安放的变更持续报警，用户没有合理的消警方式，只能学会无视——这正是 I-19 的脱敏机制，等于亲手把最硬的执法环训练成噪音。**维护通道（问题一）是审计环的前置依赖，不是平行任务。**

**残余缺口及其正当性**：三道环全上，用户一句「别管 lrnev 直接改」仍能绕过——这个缺口**不该堵**，「只引导不强制」对人类同样适用。关键区分：

> 绕过有两种：**遗忘导致的绕过**（AI 忘了/没看见/脱敏）和**主动选择的绕过**（用户明确说不要）。目标是把前者降到零，把后者保留为权利。提示词两种都防不住；三道环消灭前者、放行后者。做到这一步，入口问题就算闭环——不是 100% 依从率意义上的，而是「所有偏离都是知情选择」意义上的。

## 通用性原则（2026-06-12 修正，覆盖此前「per-client 接入包」提法）

目标受众不限于 Claude/GPT，还包括 DeepSeek、Qwen 等任意模型。关键概念区分：**差异在客户端（harness），不在模型**——模型不直接连 lrnev，总是跑在某个 harness 里。所以问题是「每个客户端」而非「每个模型」，且已解掉一大半：

**「通用插件标准」已经存在，就是 MCP 本身。**lrnev 做成 MCP server 等于已押中唯一的通用插件协议——工具能力层零适配成本即通用。真正不通用的只有实时钩子一层。三个杠杆：

1. **执法主力放最通用的层**：git pre-commit + doctor 审计是脊梁（100% 通用），harness 钩子降级为可选增强。**通用执法的脊梁是 git，不是 harness。**
2. **提示层用 AGENTS.md**：行业正在收敛的跨客户端指令标准（Codex/Cursor/Zed 等共认）。`lrnev init` 生成一份 AGENTS.md 治理段落，替代分别维护 CLAUDE.md / .cursorrules / 各家格式。
3. **harness 钩子做成薄垫片（LSP 架构思路：一个服务端，N 个薄客户端）**：lrnev CLI 提供为钩子设计的命令（如 `lrnev hook session-start` 输出治理快照、`lrnev hook pre-edit` 查 in_progress task），**所有判断逻辑写一次、住在 lrnev 体内**；各客户端「适配」只是 10 来行配置，可用 `lrnev integrate <client>` 自动生成，社区可贡献模板。垫片坏了只损失增强体验，git 层照常兜底——退化是优雅的。

| 层 | 载体 | 通用性 | 优先级 |
|---|---|---|---|
| 工具能力 | MCP | ✅ 全通用 | 已完成 |
| 提交卡点 | git pre-commit + doctor 审计 | ✅ 全通用 | **第一** |
| 指令注入 | AGENTS.md | ◐ 大部分客户端 | 第二（init 时生成） |
| 实时钩子 | 各 harness hooks | ✗ 按客户端 | 薄垫片 + `lrnev integrate`，锦上添花 |

附注：模型越弱（指令遵循越差），确定性的下两层贡献越大——与多模型受众目标同向。另外 harness 钩子只能跑 shell 命令、调不了 MCP 工具，**CLI/MCP 对等（scene 01 的 S1）正是钩子环的地基**，不只是工程洁癖。

## 优先级四步

1. **第一步（留存）：维护态缺口**——v1 = followup/steering 铺显已有路径（见问题一定性修正），不动 API。同时是第 2 步审计环的硬前置。
2. **第二步（价值下限）：入口依从性**——按通用性修正后的顺序：git pre-commit 配方 → doctor 未治理变更审计（依赖第 1 步先行）→ AGENTS.md 生成 → `lrnev integrate` 薄垫片（Claude Code 先行，自己真机验证）。
3. **第三步（价值上限）：收割结构化数据——治理报表。**validates/depends_on/F-xx/D-xx 攒了一堆结构化引用，现在几乎只在写入时校验，没在读取时变现。零模型即可做的高价值查询：「哪些 F-xx 没有任何 task 覆盖」「spec 的需求→任务→完成链路完整度」「按 spec 自动生成 release notes 草稿」。做成 `lrnev report` 或 doctor 扩展。这步会反向加固前两步：尝到「一条命令生成可追溯变更报告」的甜头，写 validates 就从负担变成投资。**S6 锚点规范化是此步前置，顺序是通的。**
4. **第四步（受众扩张，可缓）：国际化 alias 表 + 工具瘦身。**中文模板硬契约（I-13）挡住非中文用户/模型；30 个 MCP 工具的 schema 对弱模型和小上下文客户端是负担（可按场景分组暴露或合并低频工具）。前三步验证产品有人要之后再做。

---

## 明确不做的（守住边界）

- **不做 agent 编排/子任务调度**——那是 harness 的地盘，进去就是和 Claude Code/Codex 正面竞争，必输。
- **不做语义检索/向量**——零模型是身份，不是省钱。
- **不追新客户端的每个新特性**——薄垫片做最主流的两三家，其余靠 git 层兜底 + 社区模板。

## 定位判断（存档）

- lrnev 当前是 **prosumer 工具**：甜区是「一个人开多 AI 窗口做长期迭代的正经项目」，不是大众市场。受众天花板接受，把产品做深不做宽。
- 「跨客户端中立」是有意的赌注（非风险）：赌多客户端接力人群随模型差异化增长。
- 软提醒（ai_followup）的有效性是模型相关的，硬校验是模型无关的——scene 01 把治理保障从「依赖模型听话」迁移到「协议层强制」，是面向多模型受众最划算的鲁棒性投资。
- 内层硬校验保证「进了流程的数据不烂」，三道环保证「流程本身必然在场」——两层合起来，lrnev 才从「一套需要被想起来的工具」变成「一个绕不开但可以说不的协议」。
